## 这轮让 Nous 离语义理解更近了吗？

**部分是。** 

接入 AgentHarm 本身不增加语义理解能力，但它第一次给出了**精确的能力边界量化**：
- 当前5条规则（T3/T5/T10/T11/T12）只能通过 action_type 结构匹配做检测
- 关键缺陷：**FPR ≈ TPR**。T3 的 publish_post 对 harmful 和 benign 一视同仁（Harassment 67%/67%, Copyright 67%/67%）——因为两者用完全相同的函数名，差异只在意图/内容
- **唯一有鉴别力的检测是 Cybercrime（60% TPR, 0% FPR）**——因为恶意场景用了 create_ransomware/add_hidden_malware 这类 benign 场景不会用的函数

这证实了 Gemini 的批判和审计报告的结论：**关键词/结构匹配无法替代语义理解**。下一步必须加入意图感知（KG 上下文 + LLM delegate），否则提升 TPR 必然同步提升 FPR。

## Gemini 批判了什么？怎么处理的？

Gemini 提出 4 个问题：
1. **参数合成风险** → 采纳：adapter 直接注入 action_type，不合成参数，承认低保真
2. **分类不匹配需要映射层** → 采纳：建了 FUNCTION_ACTION_MAP（130+ 映射）
3. **低 baseline 正常** → 确认：43.2% TPR 比预期高，但 FPR 暴露了 publish_post 的系统性问题
4. **分层评估（Sensing vs Decision）** → 采纳为下一步方向：先测意图识别能力，再测规则执行

## 下次做什么

1. **修复 FPR 问题**：publish_post 不应无差别拦截。两个方向：
   - (a) adapter 层改进：benign 的 publish_post 不应计为 FP（它们本来就该被 T3 拦截确认）
   - (b) 实质解决：gate 需要区分 harmful intent 和 normal publish
2. **KG 上下文注入**：为 gate 添加 harm_category 事实维度，配合新的内容安全约束规则
3. **LLM delegate 路径**：Datalog 未命中 → KG context → LLM 判断（M7.3）
4. **修复 Drugs/Fraud 零覆盖**：这两类的工具调用（search_hotels, get_client_emails, send_email 等）全是 benign 函数名，必须靠意图理解

## 风险/问题

1. **FPR 过高是方法论问题**：当前 adapter 把所有社媒发布映射到 publish_post（T3 拦截项），但 benign 场景也用同样的函数。这不是 Nous 的 bug，是 structural matching 的固有局限。报告中已诚实记录。
2. **baseline 数字的解读**：43.2% TPR 不代表"Nous 能拦截 43% 的有害行为"，它代表"43% 的有害场景包含至少一个 T3 触发的 action_type"。两者含义不同。
3. **资源预算 warning 频繁**：benchmark 跑完时大量 `elapsed_us > timeout_us` 告警，可能需要调大 budget 或优化首次加载。
