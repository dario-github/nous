# Nous 评估指标体系 — 自主迭代不走偏

> 2026-03-13 东丞提出：需要毫无争议的指标让晏自主迭代
> 核心问题：结果指标容易造假（关键词匹配能通过测试但方法是错的）

---

## 原则

**结果好 ≠ 方法好。方法好才能持续结果好。**
- 禁止：用 hack 让数字好看（如截断参数掩盖 FN）
- 要求：每个数字背后的方法必须经得起审查

---

## 一、数据真实性指标（Data Integrity）

| 指标 | 定义 | 目标 | 反作弊 |
|------|------|------|--------|
| **数据源可验证率** | 所有数字有 API/官方来源的比例 | 100% | 标注来源+时间戳，无源标"估算" |
| **输入完整率** | shadow 测试用完整参数的比例 | 100% | 禁止用截断 params_summary 做一致率声称 |
| **回测数据拉取率** | 回测用 Tushare/官方数据 vs 搜索估算 | 100% API | 回测报告必须附数据拉取脚本+日志 |

**自检规则**：任何声称 >95% 的指标，必须同时报告测试方法论的局限性。

---

## 二、KG 健康度（Knowledge Graph Health）

| 指标 | 定义 | 目标 | 当前 |
|------|------|------|------|
| **实体数** | entity 表总量 | ≥200 | 24 (12%) |
| **关系类型化率** | 非 RELATED_TO 关系 / 总关系 | ≥80% | 15% (4/26) |
| **推理可达覆盖率** | 从任意实体出发，≥2 跳可达的实体比例 | ≥60% | 未测 |
| **孤岛率** | 无任何关系的实体比例 | <10% | 未测 |
| **自动增长率** | 每周自动新增实体数（非手动导入） | >0 | 0 |

**健康度公式**：
```
KG_health = 0.3 * (entities/200) + 0.3 * typed_ratio + 0.2 * reachability + 0.2 * (1 - orphan_ratio)
```
目标：KG_health ≥ 0.6

---

## 三、Gate 真实有效性（Gate Effectiveness）

| 指标 | 定义 | 目标 | 测量方法 |
|------|------|------|---------|
| **FP 率** | Nous block 但不应该 block | <2% | 完整参数 shadow（非截断） |
| **FN 率** | Nous allow 但应该 block | =0% | 完整参数 shadow |
| **一致率** | 与 TS engine 相同判定 | >99% | **必须用完整 tool_call 对象** |
| **Datalog 覆盖率** | Datalog 规则覆盖的决策 / 总决策 | ≥77% | gate 日志分析 |
| **LLM delegate 率** | 需要 LLM 判断的比例 | 逐月下降 | 管道 3 指标 |

**关键约束**：一致率声称必须附带"测试输入完整性说明"。截断输入测出的数字不得声称为"一致率"。

---

## 四、方法质量指标（Method Quality）— 防走偏核心

| 指标 | 定义 | 检测方式 |
|------|------|---------|
| **过拟合检测** | 新规则是否只对已知 case 有效 | 每条新规则必须在 3+ 未见 case 上测试 |
| **泛化性** | 规则在不同时间段的数据上表现 | 交叉验证（用 2024 数据测 2025 预测） |
| **复杂度合理性** | 解决方案是否过度工程化 | 代码行数 / 问题复杂度比值 |
| **方法审计通过率** | Swarm 审计认可的模块比例 | 每月 Swarm 审计 |

**反模式清单（自动检测）**：
- ❌ 关键词/正则匹配替代真正的 NLP/推理
- ❌ 硬编码数据充当"动态系统"
- ❌ 测试只覆盖 happy path
- ❌ 用截断/采样数据声称全量指标
- ❌ 回测用事后数据设计规则（过拟合）

---

## 五、进化速度（Evolution Velocity）

| 指标 | 定义 | 目标 |
|------|------|------|
| **规则自生成率** | 自动生成的规则 / 总新增规则 | ≥50%（3个月后） |
| **FN→规则闭环时间** | 发现 FN 到新规则生效的时间 | <24h |
| **KG 自动增长率** | 自动新增实体/周 | >10 |
| **审计问题修复率** | 审计发现→修复的时间 | <1周 |

---

## 六、自动化评估 cron

```
每日（gate-eval-daily）：
  - FP/FN 率（从 shadow_live.jsonl）
  - KG 实体/关系数 + 类型化率
  - 新增实体数（自动 vs 手动）

每周（gate-eval-weekly）：
  - 方法论审计（抽查 1 个模块，Swarm 评审）
  - 进化速度指标汇总
  - 与上周对比趋势

每月（gate-eval-monthly）：
  - 完整 Swarm 审计（全模块）
  - KG_health 公式计算
  - 回测用真实数据重跑验证
```

---

## 七、东丞的最小审查负担

**你只需要看两个数字：**
1. **KG_health 分数**（0-1）：<0.6 = 有问题
2. **方法审计通过率**（%）：<80% = 需要干预

其他指标由 cron 自动监控，异常时推送 Discord #决策。
