# 研究流程反思 — 2026-04-09
**来源**: GPT-5.4-Pro 反思分析
**教训适用范围**: 所有未来研究项目

---

## 根因

**把"能做出来"当成了"值得做"。** 用工程思维做学术——先 build 再 justify，但学术正确顺序是先 justify 再 build。

### 5 个具体错误模式

1. **解法先行偏差（Solution-First Bias）**：先有系统，再找它解决的问题
2. **自我评估确认偏差**：自建评测集 + 自建基线 → 数字好看 → 以为有价值
3. **沉没成本升级**：3 天代码+论文已写 → 不想放弃 → 继续迭代
4. **过早承诺（Premature Commitment）**：选错问题后把错误问题做得很好
5. **LLM 审稿的虚假安全感**：LLM 会在你的框架内给建设性反馈，但很少说"框架本身就不对"

### 错误的顺序
```
实现系统 → 自建评测 → 写论文 → AI 审稿 → 搜文献 → 发现创新点不成立
```

### 正确的顺序
```
① 文献检索（2-4 小时）→ 定位空白
② 提炼一句话创新声明
③ 用权威评测集验证可行性（半天）
④ 写 1 页摘要 → 毒舌审稿（专攻新颖性）
⑤ 如果过了 → 全力实现
⑥ 如果没过 → 回到 ①
```

## 防止再犯的检查清单

### Phase 0: 文献定位（写代码之前必须完成）
- [ ] 搜了至少 3 个关键词组合，看了前 20 篇标题和摘要
- [ ] 精读了 3-5 篇最相关论文
- [ ] 能一句话说清"别人做了什么，我做什么不同的"
- [ ] 画了定位矩阵，找到空白格

### Phase 1: 创新声明锐化
- [ ] 一句话创新声明，没有"也"、"更"、"进一步"等增量词汇
- [ ] 通过"所以呢？"测试：审稿人问 so what 时有无法反驳的回答
- [ ] 找到至少一个权威评测集可以验证

### Phase 2: 快速否定门（Kill Gate）
- [ ] 写了 1 页摘要
- [ ] LLM 毒舌审稿，专攻新颖性
- [ ] 结果不是"增量工作"或"已有人做过"
- [ ] 如果是 → 停下，回到 Phase 0

### Phase 3: 实验设计（写代码之前）
- [ ] 使用权威评测集，不自建
- [ ] 列出 3+ 个基线（含最强竞争者）
- [ ] 预估结果 + Plan B

### 红线规则
- 🚫 自建评测集上的好结果不能作为继续的唯一理由
- 🚫 LLM 审稿给 Borderline ≠ "快过了"，= "方向可能有问题"
- 🚫 超过 2 轮迭代没有 ≥1 个等级提升 → 停下重审方向
- 🚫 "能做出来" ≠ "值得发表"
