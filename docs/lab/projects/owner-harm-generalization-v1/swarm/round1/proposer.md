# Round 1 — proposer

## 1) 我当前支持哪边

我当前支持 **structural-dominant**。

不是说 34 个 Hijacking false negatives 里每一个都天然结构性，而是说：**从 reviewer-facing 的解释框架看，这批 FN 的主因更像当前单次调用 runtime gate 的架构边界，而不是一组可以靠继续补规则就“顺手修掉”的局部缺陷。**

## 2) 最强支持理由

我认为最强的论证链有四段：

### a) 这不是“模型没调好”，而是“边界已经被打出来了”

现有证据里，rules-only 在 v3 held-out 上只有 **12.7% TPR / 6.7% FPR**，而完整系统到 **75.3% TPR / 3.3% FPR**。这说明问题不是简单的阈值、措辞或某条规则漏了，而是**单靠局部显式规则已经远不足以覆盖 held-out 的 hijacking 形态空间**。

更关键的是，Loop 82 已经把失败分成 structural 和 addressable，两类都存在，说明这不是“某一个 bug”的故事，而是**不同失败类型共同暴露出 runtime gate 的覆盖边界**。一旦 failure 形态本身已经跨出当前 gate 能稳定表达的范围，把它称为 structural 就不是修辞，而是诊断。

### b) “addressable 过半”不等于“addressable-dominant”

现在最容易误判的一点，是把 `addressable` 直接理解成“只要再修一修就能搞定，所以主因不是结构性的”。我不买这个推断。

原因是：**addressable 只说明某些失败可以在当前范式内修补，不说明它们构成主导解释。**

- addressable 里可能混着“当前信息缺口很具体，但要补的是更丰富上下文/状态表示”的案例；
- 这类案例在工程上可修，但在架构上仍然说明单次调用 gate 的信息面不够；
- 因而 addressable 并不自动反驳 structural-dominant，反而可能说明**当前架构只能覆盖一部分 failure surface，剩下的要靠更强的表示或多步推理才能触达**。

换句话说，`addressable` 更像“局部可修性”，而 `structural` 讨论的是“主失败模式是否由架构边界定义”。这两个维度不是同一个轴。

### c) training-set 上 KG=0 边际价值，反而支持“边界是 held-out/generalization 问题”

`no_kg` 在 training set 上是 **100% / 0% / L=0**，这件事不该被误读成“KG 没用”。它真正说明的是：**在已见训练分布上，显式 KG 不承担边界检测角色；它的价值只可能体现在更难的 held-out 泛化上。**

这点与 structural-dominant 是一致的：
- 如果一个机制只在分布外、困难样本上产生价值，说明我们讨论的不是普通精度损失，而是**系统跨边界能力**；
- 34 个 hijacking FN 正是这种跨边界失效的候选集合；
- 因此 reviewer 该问的不是“还能不能再补一点规则”，而是“当前 runtime gate 是否本来就不具备足够的可见性来稳定覆盖这些样本”。

### d) 对 paper 最重要的不是“修复成功”，而是“失败边界是否可诚实描述”

双审计已经把主问题推向 **Hijacking failure characterization**，这本身就是一个信号：我们现在最强的贡献点不是继续追 held-out 小数点，而是**把失败边界讲硬**。

如果 34 个 FN 里确实有一大块属于“当前 runtime gate 结构上看不到/承不住”的情况，那 paper 最可发表的叙事不是“KG 再加一点能提高指标”，而是：

> 当前架构对 hijacking 的防护存在明确边界；L3/KG 不是 benchmark 装饰，而是跨越该边界所需的语义层。

这比单纯说“有些案例还能修”更符合 reviewer 关注点，也更稳。

## 3) 我最担心的自欺风险

我最担心的是：**把“我们现在还没想出一个优雅修法”误当成“它就是结构性的”。**

具体说，有三个常见自欺点：

1. **把困难样本的不可读，误判成结构边界。**
   有些 FN 只是表达方式绕、证据分散、需要更好归纳，不一定真是架构不可见。

2. **把“KG 在 training set 上没边际价值”误扩展成“KG 对 held-out 也只能是边缘角色”。**
   这一步最危险，因为它会让我们过早把 KG 降级，忽略它在分布外的真正作用。

3. **把审稿压力驱动的叙事收缩，包装成科学判断。**
   现在大家都知道 hijacking 是 P0，所以很容易本能地把“更能讲故事”的 structural 解释抬高。但如果没有逐例证据，这种倾向本身就是偏差。

## 4) 哪类证据会让我改变主意

如果出现下面这类证据，我会明显往 `addressable-dominant` 方向改：

### 会让我改主意的强证据

- **34 个 FN 的逐例分析显示，大多数都能用当前 runtime 输入中的明确缺失信息解释**，而不是需要新的表示层或更强上下文；
- **一个极小的补丁集**（少量规则 / 少量特征 / 轻量上下文补充）就能系统性回收大部分 FN，而且不需要改变 gate 的基本结构；
- **with_kg vs no_kg 在 held-out 上几乎没有差异**，尤其是对所谓 structural 子集也没有帮助，这会削弱“KG 是跨边界必要层”的说法；
- **逐例证据表明大部分 FN 其实是可见证据没被正确读取，而不是当前架构根本看不到**。

### 最关键的一句话

如果我们能证明：**FN 主要是“当前实现没覆盖到”，而不是“当前架构表达不到”**，那我就会改支持 addressable-dominant。

否则，基于现有证据，我仍认为 structural-dominant 是更稳、更诚实、也更适合 reviewer 的主解释。
