# reviewer-sim — Round 1

## 1) 我当前支持哪边

我当前更支持 **structural-dominant**，但只是在“更可发表”的意义上支持它：

- 如果论文要对外讲清楚这 34 个 Hijacking false negatives 的性质，**把它们解释为 runtime gate 的架构边界**，比硬把它们包装成“本来可以轻松修掉的漏检”更稳。
- 以 reviewer 视角看，`addressable-dominant` 现在更脆弱，因为它要求你证明这些失败主要是可由现有框架内局部修补的，而目前给出的材料还不足以支撑这么强的归因。
- 换句话说：**我不是说 structural-dominant 已被证明为真，而是说它目前更像一条能站得住的论文叙事。**

## 2) 最强支持理由

我最强的支持理由是：**现有证据更像“边界暴露”而不是“局部 bug”**。

具体来说：

- v3 held-out 上 Hijacking TPR 只有约 **43.3%**，这说明当前 gate 在这个子类上不是偶发失误，而是有明显系统性短板。
- 现在讨论的对象是 **34 个 false negatives**，数量已经足够让人怀疑它们不是零散异常，而是同一类机制下反复出现的失败模式。
- 你们已经在用 **structural vs addressable** 做初步划分，这本身就暗示：问题不只是“有没有一条规则没写上”，而是要先判断失败是否跨越了当前 gate 的表达能力、观测范围或决策粒度。
- 从 reviewer 角度，若没有明确的反事实干预证明“加一两条局部规则就能系统性收回这些 FN”，我会默认更保守的解释：**这些 FN 至少有相当一部分是架构层面自然产生的漏斗损失。**

## 3) 我最担心的自欺风险

我最担心的自欺风险是：**把“目前还没想出怎么修”误当成“结构上不可修”**。

这类风险有两个方向：

1. **过度结构化归因**：
   看到一批失败样本相似，就直接把它们判成架构边界，但其实只是现有分析维度不够细，或者还没找到一个足够便宜的局部修补。

2. **样本选择偏差**：
   34 个 FN 也许不是完整失败空间，而只是当前审计路径最容易暴露的一批。若它们来自某种特定提示形态、特定上下文长度、特定证据布局，那么“结构性”可能只是该切片上的结构性，不代表总体。

所以我会警惕一种很自然但危险的叙事冲动：**把高频、难修、看起来一致的失败，直接升格为架构本体论结论。**

## 4) 哪类证据会让我改变主意

要让我转向 `addressable-dominant`，我最需要的是**针对性干预后的强反事实证据**，而不是更多事后分类。

最有说服力的证据会是下面这几类：

- **局部补丁显著回收**：对这 34 个 FN 中的一个清晰子群，只做很小的 runtime gate 调整、提示约束、阈值重设或规则补丁，就能稳定大幅提升召回，而且不明显伤到别的类。
- **可重复的消融链条**：能证明这些失败并非来自架构边界，而是来自某个具体可修部件，例如检索、地址解析、证据聚合、冲突消解中的某一环。
- **跨切片一致性**：如果在不同 prompt 变体、不同上下文长度、不同表达方式下，这些 FN 大量消失，那说明它们更像脆弱实现而不是硬边界。
- **错误可解释为“遗漏一条规则”**：若大部分 FN 都能被统一归为少数几个明确的缺失规则，并且补上这些规则后问题显著缓解，那我会把它们看成 addressable。

反过来，如果后续证据显示：

- 失败样本在不同表述下依然顽固复现；
- 针对单点补丁几乎不带来净收益；
- 失败集中在 gate 本身无法表达/无法观测的场景；

那我会更坚定地支持 structural-dominant。

## 一句话结论

**当前更可发表的说法是 structural-dominant；最脆弱的是把这些 Hijacking false negatives 讲成主要是现成框架里顺手可修的 addressable failures。**
