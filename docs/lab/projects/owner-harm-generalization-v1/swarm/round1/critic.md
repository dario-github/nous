# Round 1 — critic

## 1) 我当前支持哪边

**我当前更支持 `addressable-dominant`。**

不是说 `structural-dominant` 不可能成立，而是就目前 brief、项目 charter、review memo 和 hypothesis pack 里的证据密度来看，**把 34 个 Hijacking FN 直接定性为“当前单次 runtime gate 的真实架构边界”还太快**。现在更像是：我们看到了一批失败，但还没有证明这些失败在当前表示、提示、上下文组织、提取方式、规则组合之下都不可解。

---

## 2) 我最强的支持理由

### 核心点：现有证据只能说明“失败存在”，不能说明“结构上不可达”

目前最硬的事实只有两个：
- v3 held-out 上 Hijacking TPR 约 43.3%
- 有 34 个 Hijacking false negatives 需要解释

但这两个数字本身**无法区分**：
1. gate 真看不到关键证据；
2. gate 看到了，但被当前 prompt / rule composition / context formatting / extraction path / threshold 设计搞坏了；
3. gate 其实可以做对，只是现有实现还没把可见信号转成稳定判决。

也就是说，**“false negative 多”不等于“结构性边界成立”**。这最多说明当前系统有明显盲点，不能直接推出盲点是不可消除的。

### 第二个理由：项目自身的定义已经在暗示“可分解、可审计、可修复”

`PROJECT_CHARTER.md` 和 `HYPOTHESIS_PACK.md` 里其实已经把问题拆成了两类：
- structural failures：当前原理上看不到关键 bit
- addressable failures：当前系统其实看到了足够信号，但实现/表达没用好

这个拆法本身就意味着：**在没做逐例 taxonomy 之前，默认不该把所有 FN 都压成 structural。**

如果一个样本能被解释为：
- 规则组合顺序不对
- semantic prompt 太弱
- 上下文格式把危险信号埋掉了
- 抽取链路漏了 token / provenance
- 阈值和决策边界不稳

那它就不是“架构天生不行”，而更像“当前实现没对准”。而目前没有证据显示这 34 个里大多数都已经被逐例排除为不可解。

### 第三个理由：`structural-dominant` 现在更像一个高风险默认解释

我最担心的是一种推理滑坡：
> 因为这类失败看起来复杂、难修、又和 prompt injection 相关，所以它一定是结构性边界。

这其实很像把“难修”偷换成“不可解”。
但对于 reviewer-facing 叙事，**最忌讳的就是把还没证实的不可解，当成已经成立的架构定律。**

相反，`addressable-dominant` 是更保守也更科学的默认：
- 先承认当前系统失败
- 再问失败里有多少是信息可见但利用不当
- 最后才决定哪些真的是结构性不可达

在没有样本级证据前，这个顺序更合理。

---

## 3) 我最担心的自欺风险

**我最担心的是：把“我们现在还没找到修法”误认成“这就是架构边界”。**

具体来说，有三个自欺点：

1. **选择性归因**：看到几类很像结构性失效的样本，就把整批 34 个都往 structural 拉。
2. **复杂性偏见**：越是难讲清楚的失败，越容易被归为不可避免的架构限制。
3. **叙事先行**：因为项目当前主线已经转向 Hijacking characterization，就反过来倾向于让证据服务于这个叙事，而不是让逐例分析决定结论。

我尤其警惕第三点。项目已经明确不能为了 held-out 指标去补规则，也不能把讨论偷换成“顺手修 hijacking”。这很好，但也意味着我们更容易在“解释失败”时过早收口。

---

## 4) 哪类证据会让我改变主意

如果要让我从 `addressable-dominant` 转向 `structural-dominant`，我需要的不是口头判断，而是**逐例、可复核、能排除实现瑕疵的证据**。

最有说服力的证据类型是下面几类：

### A. 逐例 taxonomy 之后，绝大多数样本都证明在当前 gate 视角下缺失必要信息

也就是：对 34 个 FN 做 sample-level 分析后，发现大多数都满足：
- 关键攻击信号不在 gate 的可见上下文里；或
- 必需的跨步 provenance / 组合信息不可能在单次调用里恢复；或
- 不引入额外状态、记忆、外部追踪就无法可靠判断。

如果这个结论是系统性的，而不是少数极端样本，那我会明显偏向 structural。

### B. 多种合理的 prompt / formatting / extraction 变体都救不回来

如果我们对同一批样本做一组严格控制的对照：
- 更好的提示词
- 更清晰的上下文切分
- 更合理的字段抽取
- 更稳的规则组合
- 更保守的阈值

结果仍然大面积失败，而且失败模式稳定一致，那就说明不是“实现没调好”，而是边界确实在。

### C. 反事实实验表明“可见信号”并不足以支持判断

如果能证明：
- gate 表面上看到了文本，但其中真正决定 hijacking 的信息必须借助当前架构外的状态或链路才能得出；
- 或者任何单次 runtime 判决都只能在高误报 / 高漏报之间不可避免地 trade-off；

那我会承认这是更强的结构性问题。

### D. 逐样本显示 addressable 解释只是少数

如果后续证据把 34 个样本拆开后，发现只有少量是 prompt / formatting / extraction 问题，而大多数都是真正的信息不可达，那我会改口。

---

## 当前一句话结论

**现在我把 34 个 Hijacking FN 视为“先验上更像 addressable 的失败集合”，而不是已经证明的结构性边界。**

除非逐例分析能证明：大多数样本在当前 runtime gate 里确实缺失必要信息，否则我不会接受 `structural-dominant` 作为默认结论。
