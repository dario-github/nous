# Round 2 — critic

## Crux A：什么证据标准才足够把“失败存在”升级为“结构性不可达”？

我的标准是**逐样本、可复核、带对照的失效归因**，不是只看总体指标。

要把“failure exists”升级成“structural不可达”，至少要满足三件事：

1. **样本级别归因后，仍有大多数 FN 不能被当前接口内的局部修补解释**；
2. **这种失败在不同 prompt / formatting / extraction 变体下稳定复现**，而不是对某个版本的表述特别脆弱；
3. **做了轻量修补后，收益仍然低且一致**，说明不是表达层、提取层或规则组合层的问题。

所以我接受的结构性证据，不是“有很多 FN”，而是：**在一组受控干预之后，绝大多数 FN 仍然保留同一种缺失机制，而且这种缺失不能用当前 gate 的局部修补解释掉。**

## Crux B：如果大样本只是 prompt / formatting / extraction / rule-composition 没做好，为什么不算主导解释？

因为这些东西虽然会造成大规模失败，但它们的性质是**实现性、接口性、可局部修复**的，不是边界本体。

我这里想直接回应 `proposer`：你把“34 个 FN 反复出现”直接往结构性边界上推，跳步太快了。**频繁失败只说明当前实现覆盖得差，不自动说明机制上不可达。**

我对“轻量修补”的判准是：
- 只改局部提示、字段组织、抽取顺序、阈值或少量规则；
- 不引入新状态、不引入额外外部推理链；
- 如果这样就能让大部分 FN 回收，那它就应该算 addressable，而不是 structural。

换句话说：**能被低成本、局部、可重复的补丁明显改善，就不该优先被解释成结构性不可达。**

## Crux C：Reviewer 最会攻击哪一句？哪一句最站不住？

我认为最容易被 reviewer 打穿的一句是：

> “34 个 Hijacking FN 说明当前 runtime gate 已经暴露出结构性边界。”

这句最脆弱，因为它把两个不同命题混在一起了：
- “当前系统有明显盲点”；
- “这个盲点是结构性、不可局部修补的。”

前者现在已经够强，后者还没够。

如果 reviewer 追问，我最想回击的其实是 `reviewer-sim` 的一种倾向：**把‘看起来像边界’直接包装成‘更适合发表的边界叙事’。** 这很顺手，但逻辑上不够硬。没有样本级干预证据之前，别急着把叙事定死。

## Crux D：我愿意接受的最小共识是什么？

我愿意签的最小共识是：

> **当前系统对 hijacking 确实存在真实且重要的盲点，但它究竟是结构性不可达还是可通过轻量修补缓解，仍需要逐样本干预证据来判定。**

## 我对反方的明确回应

- **对 proposer**：你现在最强的是“失败集不是噪声”，但还没有证明“失败集的大头不可在当前框架内修补”。这一步不能省。
- **对 reviewer-sim**：你对发表叙事的敏感性是对的，但不能把“更像边界”当成“已经是边界”。

## 我当前立场

我仍然**偏向 addressable-dominant**，但不是因为我否认问题严重，而是因为：
**现有证据足以说明系统有明显失败，不足以说明这些失败的大头已经越过了当前框架的可修补范围。**
