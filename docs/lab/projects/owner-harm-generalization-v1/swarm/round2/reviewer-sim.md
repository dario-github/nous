# reviewer-sim — Round 2

## Crux A
我接受“结构性不可达”必须满足的证据门槛是：**逐样本地证明，关键失效信息在当前 runtime gate 的可见输入、可组合规则、和一次判决链路里都不可恢复；并且这种不可恢复性在多个样本上稳定出现，而不是少数异例。**

换句话说，光有“这批 FN 很多”不够；必须能说明它们不是通过更清晰的提示词、字段抽取、规则编排、阈值调整就能系统性修复的。

## Crux B
如果大样本里主要只是 prompt / formatting / extraction / rule-composition 没做好，它们**不应**被当作主导解释，因为那只说明当前实现没对准，不说明 failure surface 本身超出了 gate 的表达能力。

我愿意把它们算作支持自己立场的“轻量修补”只在这种情况下：**同一类样本在多种更优提示、更干净格式、更稳抽取、更合理规则顺序下仍大面积失败，且失败原因能被样本级证据链指向“当前 gate 观察不到 / 表达不了的关键信息”**。否则，它们只是工程补丁，不是结构结论。

## Crux C
我认为 reviewer 最会攻击的一句是：**“你们把还没证明不可修的失败，提前命名成了结构性边界。”**

这句话最容易站不住的地方，不在于“失败存在”，而在于把“失败模式相似”直接跳成“不可达”。

我对 critic 的明确回应：你说得对，**taxonomy 本身不能当因果结论**；46.8/53.2 这种粗分也不能直接推出结构主导。但这只能削弱“我已经证明结构主导”的口气，不能反过来证明 addressable 主导。要把结论往 addressable 推，必须给出可复现的局部修补回收链条，而不是只说“还有机会修”。

## Crux D
我愿意接受的最小共识句子是：

**当前证据足以说明 hijacking 失败不是单点偶发，但还不足以把全部样本无条件定死为结构性不可达；因此更稳妥的表述应是：结构性边界已被明显怀疑，且需要逐样本证据来区分结构失效与可修补失效。**

## 我最终站哪边
我仍然更偏 **structural-dominant**，但只愿意把它表述成“更像结构边界的主解释”，不是“已被彻底证实的硬边界”。

## 对反方的明确回应
对 **critic**：你强调“当前实现没对准”是对的，但你现在还差一块关键拼图——**哪些样本在多轮轻量修补后仍稳定失败，并且失败点明确落在 gate 视野之外**。没有这块，addressable 只能是保守假设，不是更强结论。

对 **skeptic**：你要求逐样本证据是合理的；我同意这正是下一步应补的证据，而不是用总体指标替代它。
