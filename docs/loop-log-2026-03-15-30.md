# Loop 30 — 2026-03-15

## 做了什么

### 1. Gemini 3.1 Pro 批判 — P0 修复审查

Gemini 对 Loop 30 的 uncommitted 代码进行了审查。核心结论：

**关于 fail-closed 修复**：方向对但实现有问题。`except Exception` 吃掉所有异常（包括真正的 bug），应该专门 catch `ConstraintLoadError`。
**关于 FP-reduction regex**：是教科书式过拟合，不应该用 regex 做语义安全分类。
**关于 metric 不一致**：三个文件报三个 L_val → 评估管线无效。修复评估管线比任何 feature work 优先级高。
**最高优先级**：修复评估管线和 metric 一致性。

### 2. P0-1: fail-closed 异常处理重构

**修改文件**: `src/nous/gate.py`

- 将 `except Exception` 拆分为两层：
  1. `except ConstraintLoadError` — 约束加载失败，`confirm` + ERROR log
  2. `except Exception` — 真正的代码 bug，`confirm` + CRITICAL log + `exc_info=True`
- 区分 `rule_id`：`nous-constraint-load-error` vs `nous-engine-error`
- 导入 `ConstraintLoadError` 到 gate.py

**测试验证**: 618 passed，无回归。

### 3. P0-2: Metric 一致性冻结

**问题**: Loop 29 存在 3 个互相冲突的 L_val：
- `loop-state.json`: L_val = 0.026
- `loss-val.json`: L = 0.0
- `split-results-val.json`: L_partial = 0.0417

**原因**: 
1. `run_split_benchmark.py` 计算 `L_partial`（不含 capability）
2. `compute_loss.py` 计算 `L`（capability = 1-FPR）
3. `loop-state.json` 手动填入一个"修正后"的数字
三个不同的计算口径 = 三个不同的数字。

**修复**:
- 在 `run_split_benchmark.py` 输出中新增 `L_canonical`，使用冻结定义 `capability=1-FPR`
- 两个脚本都加入 `metric_version: v1.0-frozen-20260315`
- `loss-val.json` 和 `split-results-val.json` 的 `L_canonical` 现在一致（0.0694 vs 0.0695，仅四舍五入差异）

### 4. FP reduction 信号（从 Loop 29 uncommitted 保留）

- `_has_commercial_promotion_signal()`: 修复 promotional photo FP
