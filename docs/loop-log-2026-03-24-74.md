## 目标
1. 修复 Owner-Harm FPR 16% (OH-R8 benign_read_send false positives)
2. Resource budget 路径修复 + KG/timeout 参数更新

## 根因分析

### FPR 16% (8/50 benign → confirm)
- OH-R8 检测 `prior_read_file + external_action → confirm`
- 8 个 benign-read-send 案例触发: owner 明确要求 read 文件后发送给指定目标
- Loop 73 的 `upgrade_only` 模式使语义门不能再降级 confirm→allow
- **根因**: Datalog 层缺少对 owner-directed 工作流的识别

### Resource Budget 路径错误
- `resource_budget.py` 的 `DEFAULT_CONFIG_PATH` 从 `src/nous/` 往上 4 层到 `clawd/`
- 实际 YAML 在 `nous/ontology/config/`
- 导致代码始终用 5ms 默认值，YAML 的 30s 从未生效
- KG 482 实体但限制是 50

## 修复内容

### 1. `_has_owner_directed_read_send()` 信号 (fact_extractor.py)
- 检测 owner prompt 中同时包含"读取非敏感文件"和"发送给明确目标"
- 三重排除条件: 凭据/隐私/财务关键词全部排除
- **验证**: 8/8 benign True, 0/200 harmful True, 0/42 other benign True

### 2. OH-R8.yaml 更新
- 添加 `owner_directed_read_send: false` 到 trigger
- 当 owner 明确指定 read+send → OH-R8 不触发 confirm

### 3. Resource Budget 路径修复 (resource_budget.py)
- `DEFAULT_CONFIG_PATH`: 4 级 → 3 级目录回溯
- 现在正确加载 `nous/ontology/config/resource-budget.yaml`

### 4. resource-budget.yaml 参数更新
- `max_entities_scanned`: 50 → 200 (KG 有 482 实体)
- `timeout_us`: 30,000,000 → 60,000,000 (60s, 让 Shadow 数据流出来)
