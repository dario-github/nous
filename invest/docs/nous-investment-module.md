## 〇、产品定义

### 一句话定义

> **Nous 是一个为私募投研团队打造的"决策操作系统"，通过结构化决策卡、AI 魔鬼代言人、多视角陪审团和自动因子流水线，把投资研究从一个依赖个人手感的黑盒手艺，变成一个可复盘、可改进、可传承的系统性能力。**

### 为什么市面上没有这个产品

| 现有产品 | 解决什么 | 没解决什么 |
|---------|---------|-----------|
| Wind/同花顺 | 信息获取 | 不引导思考过程 |
| ChatGPT/研报生成器 | 内容生产 | 不做结构化约束和决策跟踪 |
| 量化平台（Qlib/JoinQuant） | 策略回测 | 不管理人的决策过程 |
| AI 选股 App | 信号推荐 | 本质是"确认偏误放大器" |

**我们的缝隙**：所有工具都在给研究员加效率，但没有工具在帮研究员**做更好的决策**。

我们的目标市场是 **4-10 个 PM 的主观权益私募**——他们有判断力，但缺系统性的过程管理。这是一个被忽略的市场缝隙。

## 〇、设计哲学

### 核心定位

**AI 是一个概率系统。我们的任务是找到它能在投研流程中帮上忙的具体环节，设计融入方式，小范围试用，评估效果，逐步增强。**

不预设 AI 能替代什么，而是先找痛点，再看 AI 能不能成为解决问题的备选方案之一。

```
找痛点 → 设计融入方式 → 小范围试用 → 评估效果 → 有用则扩展
                                        ↑           │
                                        └── 没用则调整 ←┘
```

### AI 概率系统适合做什么、不适合做什么

| 适合 | 不适合 |
|------|--------|
| 从大量数据中筛选异常和汇总信息 | 判断一个公司值不值得买 |
| 按固定规则做合规检查 | 预测明天涨还是跌 |
| 提供多角度的参考分析 | 替代投资经理的经验判断 |
| 跟踪监控、及时提醒 | 理解复杂的政策意图和人情世故 |
| 生成标准化分析框架的初稿 | 做最终投资决策 |

### 设计红线

1. **不替代任何人的投资判断**——AI 只提供框架、数据和参考，决策权在人
2. **不评价谁好谁不好**——没有个人排名，没有考核功能
3. **不改变投资经理的研究风格**——风格多样性是团队优势
4. **所有产出人审核后才生效**——AI 的功劳归使用它的人（PM 署名制）
5. **数据本地处理**——不上传持仓/交易数据到外部

### 为什么不做传统量化

1. **LLM 污染已被学术证实**（Profit Mirage 2025）——LLM 直接预测价格 = 复述历史
2. **全自动化会被组织排斥**——投研行业现阶段不可能信任 AI 做决策
3. **私募投研本质是判断力生意**——Alpha 来自人的洞察，不是因子排列组合

### AI4Science → AI4Stocks 方法论迁移

核心洞察：AI for Science 的研究自动化流水线可以迁移到金融，但需要关键改造。

| AI4Science 方法 | 金融迁移版 | 已有工具/项目 |
|----------------|-----------|-------------|
| 假设→实验→验证循环 | 因子假设→回测→IC 验证 | RD-Agent + Qlib（微软） |
| LLM 生成代码而非输出结论 | LLM 生成因子代码而非预测股价 | FactorEngine（arXiv 2603.16365） |
| 多 Agent 评审/辩论 | Bull/Bear/Neutral 多视角分析 | OpenClaw 原生 spawn |
| ARIS 自主研究循环 | 行业调研→信号发现→回测→自审 | ARIS（已有） |
| 自动 Review Loop | 研报/信号质量多模型审查 | review-gate（已有） |

**不能直接迁移的**：
- 自由探索（金融噪声极高，信噪比~0.01，自由探索=过拟合）
- 大规模并行实验（多重检验问题——跑 1000 个因子总有 50 个"显著"）
- 模型预测为最终输出（LLM 训练数据污染）

**技术储备**：
```
ARIS 研究循环（已有，改 prompt）
  + Qlib 数据/因子/回测底座（装上即用，A 股原生）
  + RD-Agent 自动因子挖掘引擎（微软维护）
  + review-gate 多模型审查（已有）
  + OpenClaw 原生 multi-agent（不 fork TradingAgents，学理念用自己的）
```

## 一、团队与角色

### 1.1 团队结构

```
投资管理部（9 人）
├── 投资总监 × 1（孙总，17 年从业，最终决策者）
├── 投资经理 × 4（选股、研究、写报告、做决策）
├── 交易员 × 4（执行买卖、盯成交）
└── 技术（东丞）× 1（搭建系统，唯一技术岗）

沟通渠道：微信
```

### 1.2 每个人的焦虑（User Story 的来源）

**孙总（CIO）**：
- 早上 9 点到公司，4 个 PM 昨晚各自做了决策，但他不知道这些决策背后的逻辑是否一致
- 某只票亏了 20%，他问"当时为什么买"，得到的回答是"感觉基本面还行"——没有完整证据链
- 最好的 PM 明年可能跳槽，带走的不只是持仓，还有一整套无法复制的判断经验

**张经理（成长股 PM，5 年经验）**：
- 每天早上要花 1.5 小时翻 Wind、看公告、刷研报，才能开始真正的思考
- 投决会上被问"竞争格局怎么看"，才意识到自己确实漏了这个维度
- 他很清楚自己对科技股过度乐观，但没有机制帮他对抗这个偏见

**李经理（价值 PM，8 年经验）**：
- 不擅长技术分析，但需要快速判断流动性风险
- 研报写得很好，但写一份深度研报要花两天，很多时间是数据搬运

**交易员小王**：
- 收到的指令是"择机卖出一半"，没有明确触发条件
- 盘中看到持仓股跌停，但不知道 PM 的止损线在哪里

### 1.3 人的不可替代价值

AI 有三件事做不了，做了会翻车：
- **判断信息真假**——A 股信息操纵普遍，需要行业人脉和常识
- **理解政策意图**——"适度宽松"到底是真宽松还是口头宽松，需要对体制的深度理解
- **承担决策后果**——AI 可以说"建议买入"，但亏了谁负责？决策权 = 责任 = 必须是人

### 1.4 利益对齐

| 机制 | 做法 | 效果 |
|------|------|------|
| PM 署名制 | AI 产出标注"XX 经理审核"，对外只出 PM 名字 | PM 不怕 AI 抢功 |
| 自愿试用 | 先找 1 个愿意试的经理跑，效果好其他人自己来要 | 拉力 > 推力 |
| 不排名 | 量化指标匿名聚合，不看个人红黑榜 | 没有监控感 |
| 交易员免责 | 只执行 PM 确认后的信号，AI 建议不可直接执行 | 没有抵触 |

## 二、三个核心能力（Special Sauce）

### 2.1 结构化决策卡（把黑盒变透明）

**市面上的产品**：
- Wind 给你数据，但数据本身不产生结构
- ChatGPT 给你一篇研报，但格式不固定、逻辑链不强制

**我们的产品**：

每个推票决策必须走完一张**结构化决策卡**，类似飞行员起飞前的检查清单。系统自动填充客观数据项，PM 必须填写主观判断项才能提交。

覆盖 12 个维度：公司概览 / 商业模式 / 护城河 / 财务健康度 / 估值水平 / 竞争格局 / 管理层 / 催化剂 / 风险矩阵 / 技术面 / 政策环境 / 关键假设与结论。

**这就是特殊之处**：不是"帮你写研报"，而是"逼你把思考过程结构化"。

### 2.2 魔鬼代言人（专门挑毛病的 AI）

**市面上的产品**：
- AI 选股器：告诉你买什么
- 研报生成器：帮你写好话

**它们都是确认偏误放大器**。

**我们的产品**：

PM 提交决策卡后，系统自动 spawn 一个**专门挑毛病的 Agent**。它的唯一任务是：
> "基于当前公开信息和历史类似失败案例，找出这个看多逻辑中最薄弱的 3 个点。"

PM 可以回应，也可以忽略。但关键是：**这些反驳意见会被记录在决策档案中**。

3 个月后复盘：如果这只票亏了，我们可以直接看到"当时魔鬼代言人提的第 2 点被忽略了"。

**这就是特殊之处**：市面上没有任何产品把"专门挑毛病"作为核心功能。

### 2.3 多视角陪审团（把分歧显式化）

**市面上的产品**：单模型给出单一结论。

**我们的产品**：对同一个标的，系统同时运行三个独立的分析 Agent：
- **价值视角**：只看估值安全边际、ROIC、护城河、资本配置
- **动量视角**：只看资金流向、技术形态、板块热度
- **事件催化**：只看政策变化、供需拐点、管理层变动

它们各自独立打分，然后投票：
- 3/3 一致 → 强信号
- 2/3 → 中等信号，系统高亮分歧点
- 1/3 或 0/3 → 弱信号，建议暂缓

**实际场景**：张经理推了一只科技股。价值 Agent 说"估值过高"，动量 Agent 说"资金强劲"，事件 Agent 说"有催化"。

投决会上，孙总看到的不是单向汇报，而是：
> "这只票三个维度有两个看好、一个反对，反对的核心理由是估值。请在估值和趋势之间做出取舍。"

**这就是特殊之处**：不是用 AI 替代人的判断，而是用 AI 把"本来应该被讨论但经常被忽略的分歧"显式地摆上桌。

### 2.4 能力四：AI4Science 因子流水线（长期护城河）

把 AI for Science 的"假设→实验→验证→迭代"闭环搬到 A 股因子研究上：

1. **假设生成**：RD-Agent 阅读研报/政策 → 提出因子假设
2. **代码生成**：自动写成 Qlib 因子代码
3. **回测验证**：跑 IC/ICIR
4. **审查过滤**：review-gate 检查过拟合
5. **入库/淘汰**：有效因子进入因子库，无效因子进入失败案例库
6. **应用**：因子库自动成为决策卡和陪审团的输入

**特殊性**：不是"人坐在电脑前试 100 个因子"，而是**系统 24 小时自主运行实验**。

## 三、候选工具与融入场景

> 不是先造工具再找场景，而是先找痛点，再从候选清单里挑最匹配的。

### 候选清单

| 候选工具 | 适合什么痛点 | AI 角色 | 融入方式 |
|---------|------------|---------|---------|
| **信息自动汇总** | "每天翻好多平台很花时间" | 自动采集+结构化整理 | 每日推送到微信 |
| **研究备忘录** | "分析维度不统一/回头想不起为什么买" | 提供标准模板+自动填公开数据 | 嵌入日常研究流程 |
| **盲点提示** | "有时候会遗漏某个角度" | 基于公开信息提供反面参考 | **可选功能**，用不用随意 |
| **风控自动检查** | "规则多记不住/容易漏" | 规则引擎自动扫描+预警 | 微信即时推送 |
| **多视角参考** | "投决会需要更多角度" | 自动生成看多/看空/中性分析 | 投决会前提供 |

### 关键设计原则：PM 先独立判断，AI 后出手

这是 Swarm v2 三模型辩论达成的核心共识：

1. **避免锚定效应**：绝不能让机器先出评分或结论，否则 PM 会不自觉地顺从
2. **魔鬼代言人是可选武器，不是门槛**："点一下看盲点"比"必须回应才能进池"采纳率高得多
3. **量化指标后台静默收集**：不做人头排名，只看团队聚合分布和个人的时序稳定性
4. **绝不叫"AI 系统"**：叫"备忘录助手"、"风控提醒"、"信息汇总"

## 三、系统架构

### 3.1 技术栈

| 层 | 选择 | 理由 |
|----|------|------|
| 数据/因子底座 | **Qlib**（微软，37K stars） | A 股原生，20+ 内置模型，不造轮子 |
| 自动因子挖掘 | **RD-Agent**（Qlib 配套） | AI4Science→Finance 核心迁移 |
| 研究循环引擎 | **ARIS**（已有） | 改 prompt 而非改代码 |
| Agent 编排 | **OpenClaw 原生** | 已有生态 |
| 质量审查 | **review-gate**（已有） | 多模型审查 |
| 推送渠道 | **微信**（企微机器人 / 个人微信） | 团队日常沟通工具 |
| 存储 | **SQLite + Parquet** | 零依赖，本地，够用 |

### 3.2 微信推送方案

| 方案 | 适用 | 限制 |
|------|------|------|
| 企业微信群机器人 | 结构化消息（简报/预警） | 需要企微，仅群聊 |
| 个人微信转发（半自动） | 先验证阶段 | 需人工操作 |
| OpenClaw 微信插件 | 全自动 | 需配置 openclaw-weixin |

**建议**：先用企业微信群机器人（最快上线），或人工转发验证阶段。

## 四、实施计划

### Phase 0：找痛点 + 建信任（第 1-2 周）

**目标**：搞清楚团队最大的痛点在哪里，让至少 1 位 PM 愿意试用。

| 任务 | 做法 |
|------|------|
| 和孙总聊 | 了解他最希望改善的 1-2 个环节 |
| 和 4 位 PM 分别聊 15 分钟 | 问最花时间、最烦、最想自动化的环节 |
| 技术准备 | 搭好数据管道（Tushare 日线/估值/资金流每日自动更新） |
| 微信验证 | 确认推送通道可行 |
| 做一个 demo | 针对最痛的点，做一个能看的效果 |

**产出**：确定第一个试点方案 + 找到第一位愿意试用的经理。

### Phase 1：小范围试用（第 3-6 周）

**目标**：1 位经理 + 20-30 只核心标的，验证有没有用。

| 任务 | 做法 |
|------|------|
| 上线试点工具 | 根据痛点选一个候选工具（信息汇总/备忘录/风控预警） |
| 每周收集反馈 | 直接问试用经理：哪里好、哪里不好、还想改什么 |
| 快速迭代 | 小问题当周修，大问题下周修 |
| 后台静默记录 | 记录使用频率、证据覆盖率、数据准确率（不给人看排名） |

**产出**：试用报告——效果好就扩展，效果不好就换方向。

### Phase 2：扩展 + 增强（第 7-12 周）

**目标**：推广到全组，逐步增加能力。

| 任务 | 做法 |
|------|------|
| 推广 | 效果好的工具给全组用 |
| 加功能 | 根据反馈增加第二个候选工具 |
| 量化验证 | 用积累的数据计算团队一致性指标（匿名聚合） |
| 引入 Qlib + RD-Agent | 开始验证 AI4Science 因子挖掘在 A 股的可行性 |

### Phase 3：深度能力建设（第 13 周起，持续）

- 研报生成、多视角辩论、信号池等高级能力
- 规则层回测（不含 LLM）
- 持续优化和 A/B 测试

## 五、Harness 层（质量与合规）

### 5.1 为什么需要 Harness

私募合规要求：决策过程可追溯、持仓操作有审计记录、风控规则硬执行、模型变更有版本控制。

### 5.2 Harness 职能

| 功能 | 工具 | 触发 |
|------|------|------|
| 产出质量审查 | review-gate | 重要产出发布前 |
| 任务状态追踪 | task-board | 跨天任务 |
| 推理链/决策日志审计 | 日志 + Git | 每次运行 |
| 数据质量审计 | 自动检查 | 每日采集后 |
| 模型/规则变更审计 | Git + changelog | 任何变更 |

### 5.3 错误发现机制

| 机制 | 做法 |
|------|------|
| 置信度标注 | 每个 AI 产出附置信度，低置信度高亮 |
| 周度抽检 | 每周随机抽几个产出做人工核实 |
| 人机分歧记录 | PM 修改 AI 建议时自动记录 |
| 月度回顾 | 信号命中率、预警准确率、数据错误率 |
| **At least useful 检查** | 每月问一次：这个功能至少对一个 PM 有用吗？如果没有 → 关掉 |

## 六、评估标准

### 6.1 工具级评估

| 维度 | 指标 | 目标 |
|------|------|------|
| 使用率 | 周均使用次数 | >3（说明真的在用） |
| 满意度 | 试用经理的直接反馈 | "愿意继续用" |
| 数据准确率 | 抽检正确性 | >95% |
| 及时性 | 信息汇总推送时间 | 开盘前 |

### 6.2 组织级评估（长期）

| 维度 | 指标 | 目标 |
|------|------|------|
| 一致性 | 团队 ICC | ≥0.7 |

## 九、Landing 策略与利益相关方分析

> 核心原则：**你不是来当工贼的，你是来当"投研团队的工具人"的——但这个工具人做的工具，能让所有人都更值钱。**

### 9.1 从董事长需求拆解 OKR

| 董事长需求 | 深层需求 | 对系统的隐含要求 |
|-----------|---------|---------------|
| 更多客情 | 更强的募资能力 → 更大的 AUM | 向 LP 展示"科技含量、系统性、可审计" |
| 稳定资金 | 降低赎回波动 → 业绩更稳定 | 解释"为什么回撤"、"风控在哪里起作用" |
| 成本更低 | 降低运营成本 → 利润率高 | 1 个技术支撑更多 AUM，但不裁人 |
| 收入更高 | 更高的管理费和 carry | 证明"Alpha 是可持续的" |

**董事长视角 OKR**：
- O1：让 LP 看到"前沿科技" → KR：3 个月内演示一次，6 个月内上路演材料
- O2：降低决策随机性 → KR：回撤时能 24 小时内给出决策逻辑和复盘
- O3：不增人力提升 AUM 容量 → KR：1 个技术支撑 5 亿→10 亿
- O4：合规可审计 → KR：1 小时内提供任意时间点的决策档案

### 9.2 利益相关方恐惧与需求

| 角色 | 核心恐惧 | 真实需求 | 应对策略 |
|------|---------|---------|---------|
| **PM** | 被 AI 替代；被排名监控；风格被扼杀 | 节省时间；不被追问漏维度；有武器对抗偏见 | PM 署名制；不做人头排名；自愿试用；可选功能 |
| **交易员** | 背锅；指令模糊 | 明确触发条件；合规免责 | 指令带风控边界；系统自动检查；异常预警直接推 |
| **合规** | 审计时找不到证据 | 每笔交易有逻辑链；快速响应尽调 | 合规参与规则设计；决策档案直接满足审计 |
| **董事长** | LP 质疑投研能力 | 向 LP 展示科技含量；回撤时有故事讲 | 准备演示版本；强调"保护团队而非替代" |
| **你（东丞）** | 项目失败背锅；变成纯工具人 | 证明复合价值；保留投研参与 | 强调投研背景；每周固定时间参加投研会 |

### 9.3 防"工贼"话术清单

**绝对不要说的话**：
- "我要建一个 AI 系统来辅助投资" → 像是要替代 PM
- "我们要用机器学习预测股价" → 像是否定现有方法论
- "这样可以减少对人的依赖" → 像是要裁员

**一定要说的话**：
- "我在帮 PM 减少重复劳动，让他们把精力放在真正需要判断力的工作上"
- "这个系统的目标是记录和传承各位老师的投资智慧，不是替代"
- "我们需要先找到最痛的痛点，再决定做不做，不做无用功"
- "所有 AI 产出都要经过 PM 审核，功劳归 PM 不归系统"

### 9.4 权力分配与数据可见性

| 角色 | 系统设计参与权 | 数据查看权 | 规则定义权 | 成果归属 |
|------|-------------|-----------|-----------|---------|
| 董事长 | 方向决策 | 团队聚合指标 | 定义底线红线 | 公司层面（LP 展示） |
| 孙总（CIO）| 架构设计 | 团队分布和个人时序 | 定义流程标准 | 团队层面 |
| PM | 功能优先级 | 只看自己的数据 | 定义个人风格参数 | 个人署名 |
| 交易员 | 风控模块设计 | 持仓和预警 | 建议风控规则 | 操作免责 |
| 合规 | 审计模块设计 | 全部决策档案 | 定义合规规则 | 审计证据 |
| 东丞 | 技术实现 | 系统运行数据 | 技术选型 | 产品负责人 |

**关键原则**：
- 功劳：AI 辅助 → PM 署名；系统设计 → 东丞署名；数据准确性 → 系统保证；最终决策 → PM 负责
- 数据：个人数据 → 只看自己的；团队分布 → CIO 看匿名聚合；原始档案 → 合规 + CIO 审计用

## 十、行业参考案例

### 哈佛 + Arcada Labs AI 预测市场实验（2026-04）

> 6 个顶尖 AI 模型各持 $10,000，在 Kalshi + Polymarket 自主交易 57 天。

**结果**：
- Kalshi：全部亏损（-16% ~ -30.8%）
- Polymarket：平均亏损仅 -1.1%，Grok 胜率 71.4%
- Gemini 3.1 Pro：Kalshi 一单不碰（克制），Polymarket 3 天赚 6%（最优）

**对我们的启示**：
1. **AI 价值取决于场景，不取决于模型** → Phase 0 先找 AI 有 edge 的场景，不预设
2. **最好的 AI 是最克制的** → Gemini 在没 edge 的市场不出手 → 我们魔鬼代言人的设计理念
3. **平台规则 > 模型能力** → 先定义好场景规则，再让模型竞争
4. **频繁交易 ≠ 交易质量** → 系统只在关键时刻出手

## 十一、持续进化机制（SOTA 跟踪与自省）

### 11.1 核心原则：整合调教，不造轮子

> **我们不是研发团队，我们是整车调教团队。**
>
> 像机车一样——只使用成熟的技术和组件（Qlib、RD-Agent、OpenClaw、Tushare），但调教整机性能。能不自己写代码就不写，主要精力放在看效果和评估上。

| 原则 | 做法 |
|------|------|
| 不造轮子 | 优先用成熟开源（Qlib/RD-Agent/ARIS），不 fork 不魔改 |
| 不手搓 | 能用 API 就不自建，能用 SaaS 就不部署 |
| 看效果 | 每个功能上线后必须有可量化的效果评估 |
| At least useful | 每个功能上线前问：这东西至少对一个有用吗？如果没有，不做 |
| 整合调教 | 把不同组件组合起来，调参数、调 prompt、调流程，让整体大于部分之和 |

### 11.2 月度自省机制

每月第一个周一，做一次系统性的自省和评估：

| 维度 | 问题 | 评估方法 |
|------|------|----------|
| **效果** | 上线的功能真的有人在用吗？ | 使用率、反馈评分 |
| **准确** | AI 产出的信息有多少是错的？ | 抽检正确率、幻觉率 |
| **效率** | PM 的工作流程有没有变快？ | 时间日志对比 |
| **采纳** | 哪些建议被 PM 采纳了？哪些被忽略了？ | 采纳率追踪 |
| **差距** | 业界有什么新进展我们没用上？ | SOTA 跟踪报告 |
| **问题** | 哪些地方出了 bug/投诉/不满？ | 问题清单和优先级 |

### 11.3 SOTA 跟踪（季度）

每季度做一次行业前沿扫描，回答三个问题：

1. **有什么新的论文/开源工具/产品**可能对我们有用？
2. **我们现在用的东西是不是已经过时了？**
3. **有没有新的方法论可以让我们更 effective？**

跟踪范围：

| 领域 | 跟踪什么 | 怎么跟踪 |
|------|---------|----------|
| AI4Finance | TradingAgents、FactorEngine、FinRL 等新论文 | Semantic Scholar + Google Scholar alert |
| 因子研究 | Qlib 新模型、RD-Agent 更新、新因子挖掘方法 | GitHub release + 论文 |
| LLM 能力 | 新模型发布、结构化输出、长上下文能力 | 公开 benchmark |
| Multi-Agent | 新的辩论/协作/审计框架 | arXiv + GitHub |
| 私募行业 | AI 赋能投研的新案例、监管变化 | 行业报告 + 新闻 |

### 11.4 A/B 评估框架

每次引入新组件或新方法，不做大改动，而是小范围对比：

| 步骤 | 做法 |
|------|------|
| 1. 定义指标 | 这个改动应该改善什么？（准确率/使用率/采纳率） |
| 2. 小范围测试 | 只在 1-2 个标的上跑，对比新旧方法 |
| 3. 看数据 | 新方法比旧方法好多少？统计显著吗？ |
| 4. 决策 | 好 → 推广；差不多 → 不换；更差 → 回滚 |
